线程池

基本概念和用途

当面对大量异步任务时，可以将任务提交给线程池，其通过预先创建并重复使用的一定数量的线程，避免创建过多的线程，降低系统资源消耗,减少任务的调用开销。

ThreadPoolExecutor根据corePoolSize和maximunPoolSize参数自动调整线程池的大小。

  • 当新任务通过execute(Runnable command)提交时，如果运行的线程小于corePoolSize，则创建新线程来处理请求
  • 如果运行线程等于或者大于corePoolSize，则 Executor 始终首选将请求加入队列，而不添加新的线程。
  • 如果无法将请求添加到阻塞队列，则尝试创建新的线程
  • 如果创建此线程后，线程数超出maximumPoolSize，此种情况下，任务将会被拒绝。

数量和状态变量管理
在ThreadPoolExecutor内部用AtomicInteger类型的ctl变量去维护两个概念上的参数，即有效的线程数量workerCount和线程池的运行状态，其中ctl的高3位表示运行状态，低29位表示数量。

各个状态含义如下:

  • RUNNING:接收新任务并且处理已经进入队列的任务

  • SHUTDOWN：不接收新任务，但是处理已经进入队列的任务

  • STOP:不接收新任务，不处理已经进入队列的任务，并且中断正在执行的任务

  • TIDYING: 所有任务执行完成，workerCount为0,线程池转到了TIDYING状态会执行terminated()钩子方法

  • TERMINATED：terminated()方法执行完成。

###说明
虽然Java根据响应的场景，通过Executors类提供了快速构造线程池的静态方法，
Executors类：

需要注意的是，实际项目中建议通过ThreadPoolExecutor的方式，这样能让开发人员更加明确线程池的运行规则，规避资源耗尽的风险。

  • FixedThreadPool和SingleThreadPool:允许的请求队列长度为Interger.MAX_VALUE，可能会堆积大量的请求，从而导致OOM
  • CachedThreadPool和ScheduledThreadPool:允许创建的线程数量为Interger.MAX_VALUE，可能会创建大量的线程，从而导致OOM

实际项目中相关参数可以自行设置，然后可以使用Btrace去分析线程池容量规划是否合理，再去优化相关参数。
